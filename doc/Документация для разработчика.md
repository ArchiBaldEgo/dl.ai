## 1. Назначение проекта и общая архитектура

DL.AI — это веб-приложение на Django + Django Channels (ASGI/Daphne), которое предоставляет интерфейс для общения с языковыми моделями в контексте обучения программированию. Приложение монтируется под префикс `/ai/` на том же домене, что и основной сайт `dl.gsu.by`. Собственной формы логина у приложения нет: аутентификация делегирована основному сайту через cookie `DLSID` и внешний auth API.

Архитектурные слои:

- **Django + Channels** — основное приложение (`DjangoTest/`, `ai/`). Обрабатывает HTTP API, WebSocket, админку, фоновые задачи.
- **PostgreSQL** — постоянное хранилище.
- **nginx** — внутренний реверс-прокси (`nginx/`).
- **bot/** — отдельный Node.js/Puppeteer сервис, который оборачивает бесплатный `chat.deepseek.com` в OpenAI-compatible API.

Важные особенности:

- Всё приложение работает под `/ai/`: URL, статика, куки, сессии, WebSocket.
- Внешняя авторизация выполняется на каждый HTTP-запрос через middleware.
- WebSocket тоже авторизуется, но основная проверка идёт по куке `DLSID` при подключении.
- Есть единый регистр клиентов моделей (`ai/model_clients/registry.py`), через который consumer и health-checker выбирают провайдера.
- Все запросы к моделям записываются в `AIRequestLog`.

## 3. Регламент работы с git

Проект использует простой git-flow для студентов и разработчиков:

1. Каждый разработчик работает в отдельной ветке, названной по фамилии.
2. По готовности открывается Pull Request в `main`.
3. PR проверяется автоматически и вручную.
4. После одобрения изменения мерджатся в `main` и выкатываются на прод.

Перед началом работы новый участник должен отправить письмо на `vadik2005guryanov@gmail.com` с:

- своим GitHub nickname;
- своей почтой для связи.

Рекомендации:

- Не коммитьте `staticfiles/`, `.env`, локальные базы SQLite, бинарные артефакты.
- Делайте атомарные коммиты с понятными сообщениями на русском или английском.
- Перед PR убедитесь, что `python manage.py test ai` проходит (или хотя бы не падает из-за ваших изменений).

## 4. Локальная разработка без Docker

### Требования

- Python 3.11+
- PostgreSQL 14+
- `psql`

### Подготовка базы

```sql
CREATE USER dlaibd WITH PASSWORD 'dlaibd';
CREATE DATABASE dl_ai OWNER dlaibd;
```

### Подготовка окружения

```bash
cp .env.example .env
# Отредактируйте .env: DB_HOST=127.0.0.1, DB_PASSWORD, SECRET_KEY и токены моделей
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python manage.py migrate
python manage.py runserver 0.0.0.0:8000
```

Приложение будет доступно по `http://127.0.0.1:8000/ai/...`. Статика отдаётся через Django в режиме разработки по `/ai/assets/`.

### Без внешнего auth API

Для локальной разработки без доступа к `dl.gsu.by` можно использовать тест-панель входа по паролю (`/ai/test-panel/login/`). Требуется предварительно создать пользователя и установить пароль через Django shell или тест-панель.

## 6. Аутентификация и авторизация

### Внешняя авторизация

Приложение не хранит паролей пользователей. Вход происходит на основном сайте `dl.gsu.by`, после чего браузер получает cookie `DLSID`. `ExternalAuthMiddleware` (`ai/middleware.py`) на каждый запрос:

1. Берёт значение куки `DLSID`.
2. Отправляет POST-запрос на `EXTERNAL_AUTH_API_URL` (по умолчанию `https://dl.gsu.by/restapi/get-user-info`) с телом `{"sessionId": "...", "removeHtmlTags": True}`.
3. Если API возвращает 401 — перенаправляет на `EXTERNAL_AUTH_REDIRECT_URL`.
4. Если API недоступен — возвращает 503 или пропускает в зависимости от настроек.
5. При успехе вызывает `get_or_create_user_from_external` (`ai/external_account.py`), который:
   - создаёт или обновляет Django `User`;
   - создаёт/обновляет `ExternalDLAccount` с `external_user_id`;
   - подбирает username из `login`, `username`, `nickname` и т.д.;
   - добавляет пользователя в группу `prompt_developer`;
   - привязывает Django-сессию к пользователю.

### Роли и права

- **prompt_developer** — группа, в которую попадают все пользователи автоматически. Даёт доступ к:
  - `/ai/admin/` (кастомная админка);
  - ARM `/ai/admin/arm/find-error/`;
  - "Моим препромптам" `/ai/admin/prompts/my/`;
  - разделу "All prompts" `/ai/admin/ai/prompt/` (только просмотр и редактирование своих/назначенных).
- **staff / superuser** — полный доступ ко всей админке, включая:
  - управление справочниками (`ProgrammingLanguage`, `Topic`, `Prompt`, `SharedPrompt`);
  - страницу состояния моделей `/ai/admin/arm/models/`;
  - логи запросов `/ai/admin/ai/airequestlog/`;
  - изменение `owner` и `editors` в промптах.

### Prompt ACL

- Каждый `Prompt` и `SharedPrompt` имеет `owner` и многие-ко-многим `editors`.
- prompt_developer видит все промпты, но редактировать может только свои или те, где он в `editors`.
- Админ может менять `owner` и `editors`.
- При создании промпта разработчиком `owner` = текущий пользователь, а сам пользователь автоматически добавляется в `editors`.

## 7. Модели данных

### ExternalDLAccount

Связь между Django `User` и внешним ID из `dl.gsu.by`:

- `user` — OneToOne к `User`;
- `external_user_id` — уникальный ID основного сайта;
- `external_login`, `external_first_name`, `external_last_name`, `external_middle_name`.

### ProgrammingLanguage

Справочник языков программирования. Поле `language_name` — отображаемое имя.

### Topic

Тема, привязанная к языку программирования. Локализованные имена: `topic_name`, `topic_name_en`, `topic_name_fr`.

### SharedPrompt

Общий (не привязанный к теме) препромпт. Может быть ограничен на конкретные языки программирования через `programming_languages`. Поля:

- `prompt_name` / `_en` / `_fr` — имя для выбора;
- `prompt_text` / `_en` / `_fr` — текст препромпта;
- `owner`, `editors`;
- `mode` — системный режим: `chat`, `solve`, `find_error` (уникальный, nullable). Системные препромпты не отображаются в выпадающем списке пользователя, но используются как fallback для режимов "Реши задачу" и "В чём ошибка".

`get_effective_text(ui_language, programming_language_name)`:

- выбирает локализованный текст по `ui_language` (`ru`, `en`, `fr`);
- подставляет плейсхолдеры:
  - `{language}` / `{язык}` — язык программирования;
  - `{topic}` / `{тема}` — тема;
  - `{message}` — текст задачи/сообщения;
  - `{code}` — код программы.

### Prompt

Препромпт, привязанный к теме. Может ссылаться на `SharedPrompt` или переопределять текст через `prompt_text_override`. Тоже имеет локализованные имена и тексты, `owner`, `editors`.

### AIAppSettings

Singleton-модель с полем `is_enabled`. Если `False`, пользовательские страницы и WebSocket блокируются с сообщением "AI app is disabled".

### AIRequestLog

Центральная модель логирования. Поля включают:

- `source` — `websocket` или `arm`;
- `mode` — `chat`, `solve`, `find_error`, `arm`;
- `external_user_id`, `username`, `user_full_name`;
- `programming_language_name`, `topic_name`, `prompt_name`;
- `model_names` (JSON массив);
- `sent_at`, `received_at`, `duration_seconds`;
- `message`, `response_text`, `error_message`;
- `tokens` (JSON объект), `status`;
- `client_id`.

### AIModelHealthRun / AIModelAvailability

Хранят результаты ежедневной проверки доступности моделей.

## 8. WebSocket: чат, решение задач, поиск ошибок

### Подключение

WebSocket endpoint: `/ai/chat/ws/<client_id>/`. Consumer: `MyConsumer` (`ai/consumers.py`).

При подключении:

1. Проверяется `AIAppSettings.is_enabled`.
2. Проверяется аутентификация либо по `request.user`, либо по куке `DLSID` через `fetch_external_user_info`.
3. Создаётся/восстанавливается `conversation_history` для данного `client_id`.

### Формат входящих сообщений

JSON с обязательными полями:

- `type` — `1` (чат), `2` (решить задачу), `3` (найти ошибку);
- `message` — текст от пользователя;
- `language` — UI-язык (`ru`, `en`, `fr`);
- `value` — ключ выбранной модели;
- `progLng` — язык программирования;
- `topic` — тема (строка);
- `preprompt` — ID выбранного препромпта или `null`;
- `code` — код программы (для `type=3`).

### Обработка

1. `clear_context` — если пришёл специальный запрос, история диалога сбрасывается.
2. `getPromptText` — формирует итоговый текст запроса:
   - для `type=2` используется `SharedPrompt` с `mode=solve` или fallback-шаблон;
   - для `type=3` используется `SharedPrompt` с `mode=find_error` или fallback-шаблон, в который подставляется `{code}`;
   - при смене UI-языка добавляется инструкция `get_language_instruction`.
3. `model_key` нормализуется через `_LEGACY_ALIASES` (`ai/consumers.py`), затем вызывается `registry.handler(model_key, prompt_text, conversation_history)`.
4. Создаётся `AIRequestLog` со статусом `pending`.
5. После получения ответа обновляются `received_at`, `duration_seconds`, `response_text`, `tokens`, `model_names`, `status`.
6. Ответ отправляется клиенту через WebSocket.

### Обработка ошибок

При исключении в процессе вызова модели:

- `humanize_model_error` формирует понятное сообщение и техническую деталь;
- пользователю отправляется `Ошибка при обработке запроса: <friendly>`;
- в `AIRequestLog.error_message` сохраняется техническая деталь;
- статус устанавливается в `error`.

## 9. Клиенты моделей

### Регистр

`ai/model_clients/registry.py` — центральная точка выбора клиента. Каждый клиент зарегистрирован под ключом, например:

- `DeepSeek_R1_Distill_Llama_70B`, `DeepSeek_V3_1`, `DeepSeek_V3_2` и др. — SambaNova;
- `Web_DeepSeek`, `Web_DeepSeek_Thinking` — бесплатный DeepSeek через bot pool;
- legacy-алиасы: `DeepSeek_R1`, `GigaChat` и др.

Ключ модели, приходящий от фронтенда, может быть legacy-алиасом; consumer нормализует его перед вызовом регистра.

### SambaNova

`ai/model_clients/sambanova.py` шлёт POST на `https://api.sambanova.ai/v1/chat/completions` с токеном `SC_TOKEN` (или `SAMBANOVA_API_KEY`). Поддерживает `conversation_history`. Возвращает `(text, completion_tokens)`.

### Web DeepSeek

`ai/model_clients/web_deepseek.py` обращается к `BOT_POOL_URL/api/send` (по умолчанию `http://localhost:3000`). Повторяет попытки при 503/504.

### Официальный DeepSeek API

Если задан `DEEPSEEK_API_TOKEN`, Django использует официальный API DeepSeek и обходит bot pool.

### Ошибки

`ai/model_clients/exceptions.py` содержит:

- базовые исключения (ModelError, ModelAuthError и т.д.);
- `humanize_model_error(raw_text, include_detail=False)` — приводит технические ошибки к человекочитаемому виду.

## 10. ARM: сравнительное тестирование моделей

ARM (multi-model check) позволяет отправить одну задачу сразу в несколько моделей и сравнить ответы.

### URL

- `/ai/admin/arm/find-error/` — форма и результаты;
- `/ai/admin/arm/find-error/start/` — POST JSON, запускает прогон;
- `/ai/admin/arm/find-error/status/?run_id=...` — GET, статус прогона.

### Права

Доступен staff/superuser и prompt_developer.

### Построение запроса

`_build_find_error_message` (`ai/admin/arm.py`) использует `SharedPrompt` с `mode=find_error`. Если системный препромпт не найден — используется fallback-шаблон.

### Выполнение

`start_arm_sequential_run` (`ai/arm_runner.py`) создаёт фоновый `threading.Thread`. Модели проверяются последовательно, результаты накапливаются в in-memory `_jobs`. Для каждой модели создаётся отдельная запись `AIRequestLog` с `source=arm`, `mode=arm`.

Job-статус хранится 6 часов, затем чистится.

### Результаты

После завершения формируется краткий отчёт (всего моделей, успешно/с ошибкой, суммарно токенов, самая быстрая модель) и карточки ответов. Каждая карточка кликабельна и открывает модальное окно с полным ответом.

## 11. Административный интерфейс и права доступа

### ai_admin_site

Кастомный админ-сайт (`ai/admin/site.py`):

- имя `admin`;
- заголовок "AI Admin";
- главная страница `/ai/chat/`;
- монтируется по `/ai/admin/`.

### Права доступа

- `can_access_admin` — staff/superuser или prompt_developer.
- `can_access_arm` — staff/superuser или prompt_developer.
- `can_access_model_status` — только staff/superuser.
- `can_access_prompt_admin` — staff/superuser или prompt_developer.
- `can_access_logs` — только staff/superuser.

### Скрытие меню

`AIAdminSite.each_context` скрывает из бокового меню модели, к которым у пользователя нет доступа, и добавляет флаги `show_arm_link`, `show_model_status_link`, `show_logs_link` для кастомных ссылок.

### Основные кастомные разделы

- `/ai/admin/ai/prompt/` — стандартный changelist промптов.
- `/ai/admin/prompts/my/` — фильтр "мои промпты".
- `/ai/admin/arm/find-error/` — ARM.
- `/ai/admin/arm/models/` — состояние моделей.
- `/ai/admin/ai/airequestlog/` — логи запросов.
- `/ai/admin/set-password/` — установка пароля для тест-панели.

## 19. Обработка кодировок в DL API клиенте

Файл: `ai/dl_api_client.py`.

Внешний DL API (`dl.gsu.by`) возвращает условия задач и сопутствующие поля в непредсказуемых кодировках. Исторически встречаются три повреждения:

1. **Корректный UTF-8** — просто декодируем.
2. **UTF-8, ошибочно декодированный как cp1251** — классическая мojibake, при которой последовательности вроде `\xD0\x9F` превращаются в пару символов `Рџ`.
3. **CP866, представленный через cp1251 codepoints** — для некоторых PDF-страниц `dl.gsu.by` извлекает русский текст так, что байты CP866 интерпретируются как cp1251 Unicode-символы (например, байт `0x8F` становится `U+040F` вместо `U+041F`).

Поэтому `_decode_response_json` не декодирует JSON одним вызовом `content.decode("utf-8")`, а строит несколько кандидатов и выбирает лучший по эвристике читаемости `_quality`.

### 19.1. Путь данных

```
фронтенд -> /ai/api/task-info/ -> views.get_task_info_view
                                  -> fetch_task_info(node_id, session_id)
                                     -> _dl_request("GET", "/restapi/get-task-info")
                                     -> _decode_response_json(response)
                                        -> выбор лучшего JSON-кандидата
                                        -> _repair_response_strings(data)
                                           -> рекурсивный ремонт строковых полей
```

То же самое происходит для `fetch_task_solution` (`/restapi/get-solution`).

### 19.2. Почему нельзя чинить «целиком»

В одном ответе может быть одновременно:

- поле `name`, записанное корректным UTF-8 кириллицей (`"Прибытие короля"`);
- поле `statement`, где CP866 байты превращены в cp1251 codepoints (`"XXVII Љ®¬­¤­л© ..."`).

Если применить преобразование `cp1251 -> cp866` ко всей строке JSON целиком, корректное `name` будет испорчено (например, `Прибытие` станет `╧Ёшс√Єшх`). Поэтому repair делается **после** парсинга JSON и **по отдельным строковым значениям**.

### 19.3. Алгоритм `_decode_response_json`

1. **UTF-8 кандидат**.
   `content.decode("utf-8")`. Если удаётся, добавляем в список кандидатов.
2. **Проверка UTF-8-as-cp1251**.
   Если UTF-8-кандидат содержит сигнатуру `lead_char + continuation_char` (например, `Р` за которым идёт символ из cp1251-диапазона `0x80-0xBF`), пробуем «откатить» его: `text.encode("cp1251").decode("utf-8")`. Результат тоже добавляем.
3. **cp1251 кандидат**.
   `content.decode("cp1251", errors="replace")` — страховка, если настоящая кодировка ответа была cp1251.
4. **Выбор лучшего кандидата**.
   Для каждого кандидата вычисляется `_quality`. Больше = лучше. В score входят:
   - `+2` за каждую кириллическую букву;
   - `+1` за латинские буквы, цифры, пробелы и пунктуацию;
   - `-3` за каждый UTF-8-as-cp1251 lead/continuation пар;
   - `-2` за «подозрительные» высокие символы (не кириллица и не обычная пунктуация).
5. **Парсинг JSON**.
   Лучший кандидат парсится через `json.loads`.
6. **Рекурсивный почин строковых полей**.
   `_repair_response_strings` обходит dict/list. Для каждой строки, имеющей сигнатуру CP866-via-cp1251, вызывается `_repair_cp866_via_cp1251`.

### 19.4. CP866-via-cp1251: что это и как чинится

CP866 (кодовая страница DOS для русского языка) и cp1251 (Windows-1251) — разные таблицы.

- В CP866 байт `0x8F` означает русскую заглавную букву **П** (`U+041F`).
- В cp1251 байт `0x8F` означает **Џ** (`U+040F`, кириллическую заглавную дже).

Если PDF-экстрактор выдаёт байты CP866, но система дальше интерпретирует каждый байт как cp1251, мы получаем в JSON не `П` (`U+041F`), а `Џ` (`U+040F`). При этом:

- ASCII-символы (`0x00-0x7F`) не страдают.
- Большинство русских букв **оказываются именно теми cp1251 символами**, которые стоят на тех же byte-значениях, что и CP866 буквы.
- Поэтому обратное преобразование возможно: сопоставить Unicode-символу его cp1251 байт, а затем декодировать этот байт как CP866.

Пример реального символа из `curl.txt`:

| Символ в ответе | Unicode | cp1251 байт | CP866 декод | Исправлено |
|-----------------|---------|-------------|-------------|------------|
| `Џ`             | U+040F  | `0x8F`      | `П`         | да         |
| `Ё`             | U+0401  | `0xA8`      | `И`         | да         |
| `ў`             | U+045E  | `0xA2`      | `В`         | да         |
| `Ґ`             | U+0490  | `0xA5`      | `Е`         | да         |
| `¬`             | U+00AC  | `0xAC`      | `М`         | да         |
| `­`             | U+00AD  | `0xAD`      | `Н`         | да         |

### 19.5. Почему repair делается посимвольно

Ранее пробовалось преобразование всей строки через `text.encode("cp1251").decode("cp866")`. Оно не работает, когда внутри строки присутствуют:

- настоящие кириллические символы, не отображающиеся в cp1251 (вызывают `UnicodeEncodeError`);
- символы Unicode replacement `U+FFFD` (тоже нельзя закодировать cp1251);
- cp1251 символы из диапазона `0x80-0x9F`, которые в CP866 не являются буквами (например, box-drawing символы).

Поэтому `_repair_cp866_via_cp1251` обрабатывает каждый символ отдельно:

- `U+FFFD` (`chr(0xFFFD)`) — удаляется.
- ASCII (`<= ""`) — копируется без изменений.
- Символ, присутствующий в cp1251 — откатывается к байту и декодируется как CP866.
- Всё остальное — копируется без изменений.

### 19.6. Детекция CP866-via-cp1251: `_looks_like_cp866_as_cp1251`

Для каждой строки считается:

- `non_ascii` — количество символов с кодом выше `0x7F`.
- `artifact_count` — количество таких символов, которые лежат в cp1251 high-byte диапазоне `0x80-0xBF` и при этом не являются легитимной кириллической пунктуацией (`ё`, `–`, `—`, `«`, `»`, кавычки и т.п.).

Строка считается повреждённой, если `artifact_count >= 5` и более 30% non-ASCII символов — это артефакты.

Эмпирический порог `0.3` выбран потому, что в нормальном русском тексте cp1251 high-byte символы встречаются редко, а в CP866-via-cp1251 — почти все гласные/согласные попадают именно туда.

### 19.7. Детекция UTF-8-as-cp1251: `_looks_like_utf8_as_cp1251`

Здесь ищется другая мojibake: последовательность UTF-8 была декодирована как cp1251. При этом:

- UTF-8 leading byte `0xD0` в cp1251 становится `Р`, `0xD1` — `С`, ..., `0xDF` — `Я`.
- UTF-8 leading byte `0xE0` — `а`, `0xE1` — `б`, ..., `0xEF` — `п`.
- UTF-8 continuation byte (`0x80-0xBF`) декодируется в один из символов `_CP1251_CONTINUATION_CHARS`.

Метод считает пары `lead_char + continuation_char`. Если таких пар ≥ 3 и они составляют больше половины всех кириллических символов, строка признаётся UTF-8-as-cp1251.

### 19.8. `_quality`: эвристика читаемости

Вызывается для каждого кандидата. Считает:

| Что | Вес |
|-----|-----|
| кириллическая буква | `+2` |
| латинская буква | `+1` |
| цифра | `+1` |
| пробел | `+1` |
| пунктуация | `+1` |
| UTF-8-as-cp1251 пара | `-3` |
| «подозрительный» high-byte символ | `-2` |

Таким образом, корректный русский текст набирает высокий score, а повреждённые строки — низкий, и `max(candidates, key=_quality)` отдаёт предпочтение человекочитаемому варианту.

### 19.9. Пример из реального трафика

Файл `curl.txt` содержит захваченный ответ `/restapi/get-task-info` для задачи 221905.

После обработки:

- `result["taskId"]` = `221905`;
- `result["name"]` = `"Прибытие короля"`;
- `result["statement"]` начинается с:

  XXVII Командный чемпионат школьников Санкт-Петербурга по программированию

3. Для строк, похожих на CP866-via-cp1251, проверить `_looks_like_cp866_as_cp1251(text)`.
4. Проверить `_repair_cp866_via_cp1251(text)`.
5. Если результат всё ещё некорректен — посмотреть, какой байт какому Unicode-символу соответствует, и дополнить `_CP1251_UNICODE_TO_BYTE` не потребуется (он строится автоматически для всех 256 байт), но может потребоваться скорректировать `_looks_like_cp866_as_cp1251` или `_quality`.

### 19.12. Предостережения

- **Не применяйте `text.encode("cp1251").decode("cp866")` ко всей строке.** Это ломается на `U+FFFD` и на смешанных полях.
- **Не делайте repair без проверки `_looks_like_cp866_as_cp1251`.** Иначе валидный UTF-8 русский текст будет испорчен.
- **Не изменяйте `_quality` без прогона существующих тестов.** Изменение весов может привести к выбору неправильного кандидата.
- **`curl.txt` и подобные fixture-файлы должны лежать в корне репозитория**, но не коммититься как production-данные без необходимости. Для тестов они необходимы.
