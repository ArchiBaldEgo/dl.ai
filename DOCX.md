## Инструкция для пользователя

1. Назначение инструкции
Инструкция предназначена для обычного пользователя AI-интерфейса. Она объясняет, как войти в раздел /ai/, выбрать режим работы, отправить сообщение, работать с задачами по программированию, искать ошибки в коде и пользоваться голосовым режимом.
Документ написан так, чтобы пользователь понимал не только порядок нажатий, но и смысл каждого режима. Это снижает число ошибочных запросов и повышает качество ответов модели.
Важно. Обычный пользователь не управляет моделями, справочниками и настройками AI. Он работает только с доступными на текущую дату функциями интерфейса.

2. Условия доступа и авторизация
Для входа пользователь должен быть уже авторизован на основном сайте. Встроенного логина в AI-интерфейсе нет: авторизация передается из основного сайта.
    1. Выполнить вход на основном сайте.
    2. Перейти в раздел /ai/.
    3. Проверить наличие параметра uid в адресной строке, например ?uid=123.
    4. Если uid отсутствует, вернуться к корректной ссылке из основного сайта или обратиться в поддержку.
Условие	Зачем нужно	Что будет при нарушении
Пользователь авторизован на основном сайте	AI-интерфейс не содержит отдельной формы логина.	Доступ невозможен или некорректен.
В URL есть uid	uid связывает сессию с пользователем/клиентом.	Сообщение 'UID query parameter is required'.
Доступна хотя бы одна модель	Модель нужна для ответа.	Сообщение 'Сегодня нет доступных моделей'.

3. Общая структура интерфейса
Интерфейс состоит из верхней панели, центральной области и правой боковой панели. Верхняя панель отвечает за язык, модель и режим. Центральная область содержит ленту сообщений и форму отправки. Правая панель может содержать ссылку на админ-панель, но она доступна только администраторам или тестерам.
Зона интерфейса	Элементы	Назначение
Верхняя панель	Язык общения, модель, режим.	Определяет, как будет сформирован запрос и на каком языке будет ответ.
Центр	Лента сообщений и форма отправки.	Основная рабочая область диалога.
Правая панель	Ссылка на админ-панель при наличии прав.	Служебная навигация для расширенных ролей.

4. Режим 'Чат с DLAI'
Режим 'Чат с DLAI' используется для свободного диалога с моделью. Он подходит для вопросов, объяснений, идей, уточнений и обсуждения материалов, когда не требуется специальная форма задачи.
    • Поле ввода принимает обычный текст.
    • Кнопка Send отправляет сообщение.
    • Enter отправляет сообщение, Shift+Enter переносит строку.
    • Ответы отображаются в виде аккордеона: последняя реплика раскрыта автоматически.
    • Clear Context очищает историю диалога и серверный контекст для текущего client_id.
Когда нажимать Clear Context. Если модель начала учитывать старые сообщения, которые больше не относятся к новой задаче, очистка контекста помогает начать новый разговор без 'хвоста' предыдущей темы.

5. Режим 'Реши задачу'
Режим предназначен для решения задач по программированию. В отличие от свободного чата, пользователь дополнительно выбирает язык программирования, тему и, при необходимости, препромпт. Без выбранного языка программирования отправка блокируется.
Поле	Обязательность	Как влияет на запрос
Язык программирования	Обязательное	Добавляет контекст 'реши задачу на языке ...'.
Тема	Зависит от языка	Уточняет учебный или технический раздел.
Препромпт	Необязательный	Задает стиль и глубину решения.
Текст задачи	Обязательный по смыслу	Содержит условие, которое нужно решить.

Фактически система преобразует сообщение пользователя в более структурированный запрос: 'реши задачу на языке ...' плюс выбранный препромпт. Поэтому качество выбранных полей влияет на качество ответа.
Рекомендация. Формулируйте задачу полностью: входные данные, выходные данные, ограничения, примеры и требуемый формат ответа.

6. Режим 'В чем ошибка?'
Режим предназначен для поиска ошибки в коде. Пользователь может указать условие задачи, код программы, язык программирования, тему и препромпт. Должно быть заполнено хотя бы одно поле: задача или код.
Поле	Что писать	Почему важно
Задача	Условие, ожидаемое поведение, примеры.	Модель понимает, что программа должна делать.
Код программы	Фрагмент или полный код с ошибкой.	Модель анализирует фактическую реализацию.
Язык программирования	Язык кода.	Без языка отправка блокируется.
Тема и препромпт	Контекст и формат разбора.	Помогают получить учебное объяснение, а не просто исправление.

Лучший запрос. Добавьте не только код, но и что именно пошло не так: ошибка компиляции, неверный ответ на тесте, зависание, превышение времени или неожиданное поведение.

7. Выбор языка интерфейса и ответа
Переключатель языка меняет подписи, плейсхолдеры и подсказки интерфейса. Кроме того, выбранный язык передается модели как инструкция отвечать на этом языке.
Язык	Что меняется	Практический эффект
Русский	Русские подписи и ответы.	Удобно для русскоязычного учебного разбора.
English	Английские подписи и ответы.	Полезно для терминологии и международных материалов.
Français	Французские подписи и ответы.	Подходит для франкоязычной аудитории.

8. Голосовой режим
Голосовой режим открывает панель управления голосовым вводом и озвучиванием ответа. Он работает в браузерах с поддержкой Web Speech API, обычно Chrome или Edge.
    • Голосовой ввод позволяет надиктовать сообщение.
    • Озвучить ответ включает воспроизведение ответа модели.
    • Стоп прекращает озвучивание.
    • Опция озвучивания think-блоков управляет проговариванием служебных рассуждений, если они доступны в интерфейсе.
    • В режиме 'В чем ошибка?' голосовой ввод заполняет поле 'Задача'.
Ограничение браузера. Если голосовые функции не работают, сначала проверьте браузер, разрешение на микрофон и поддержку Web Speech API.

9. Работа с контекстом и форматированием кода
Контекст диалога помогает модели помнить предыдущие сообщения. Это удобно при продолжении одной темы, но мешает при переходе к новой задаче. Для начала нового обсуждения следует использовать Clear Context.
Код рекомендуется оборачивать в тройные обратные кавычки. Это делает отображение читабельным и снижает риск, что модель перепутает код с обычным текстом.
```python
for i in range(5):
    print(i)
```

10. Практические примеры запросов
Ситуация	Пример хорошего запроса	Почему это хорошо
Свободный чат	Объясни простыми словами, чем отличается список от кортежа в Python.	Есть тема и ожидаемый стиль объяснения.
Реши задачу	Дана строка. Нужно посчитать количество гласных. Язык Python. Покажи алгоритм и код.	Указаны входные данные, цель и язык.

Формат ответа	Сначала кратко укажи ошибку, затем объясни причину и дай минимальное исправление.	Модель получает структуру ответа.

# Инструкция для пользователя

Этот документ описывает интерфейс обычного пользователя (без административных прав).

## 1. Как попасть в интерфейс

- Переходите в раздел /ai/ с уже выполненной авторизацией на основном сайте.
- В адресной строке обязательно должен быть параметр uid (например, ?uid=123). Без него доступ запрещен.
- Встроенный логин в этом интерфейсе отсутствует — авторизация идет через основной сайт.

## 2. Общая структура интерфейса

- Верхняя панель:
  - Язык общения (Русский / English / Français).
  - Модель (если доступна на текущую дату).
  - Режим: Чат с DLAI / Реши задачу / В чем ошибка?
- Центр:
  - Лента сообщений.
  - Форма отправки.
- Правая боковая панель:
  - Ссылка на админ-панель (доступна только администраторам/тестерам).

## 3. Режим "Чат с DLAI"

- Назначение: свободный диалог с моделью.
- Поле ввода: текстовое сообщение.
- Отправка:
  - Кнопка Send.
  - Enter отправляет, Shift+Enter делает перенос строки.
- Clear Context:
  - Полностью очищает историю диалога и серверный контекст для текущего client_id.
- Ответы показываются в виде аккордеона:
  - Кнопки Показать/Скрыть для каждой реплики.
  - Последняя реплика раскрыта автоматически.
- Форматирование кода:
  - Оборачивайте код в ``` ... ``` для читабельного отображения.

## 4. Режим "Реши задачу"

- Назначение: решение задач по программированию.
- Дополнительные поля:
  - Язык программирования (обязателен).
  - Тема (зависит от языка).
  - Препромпт (необязателен, зависит от темы).
- Без выбранного языка программирования отправка блокируется.
- Сообщение превращается в запрос: "реши задачу на языке ..." + выбранный препромпт.

## 5. Режим "В чем ошибка?"

- Назначение: поиск ошибки в коде.
- Поля:
  - Задача (условие).
  - Код программы.
  - Язык программирования (обязателен).
  - Тема и препромпт (опционально).
- Должно быть заполнено хотя бы одно поле: задача или код.
- Сообщение превращается в запрос: "найди ошибку в коде ..." + препромпт.

## 6. Язык интерфейса

- Переключатель языка меняет подписи, плейсхолдеры и текст подсказок.
- Также влияет на запрос к модели (модель получает указание отвечать на выбранном языке).

## 7. Голосовой режим

- Кнопка "Голосовой режим" открывает панель управления:
  - Голосовой ввод.
  - Озвучить ответ.
  - Стоп.
  - Озвучивать think-блоки.
- Работает в браузерах с поддержкой Web Speech API (обычно Chrome/Edge).
- В режиме "В чем ошибка?" голосовой ввод заполняет поле "Задача".

## 8. Частые проблемы

- "UID query parameter is required" — отсутствует uid в URL.
- "Сегодня нет доступных моделей" — список моделей пуст (проверка еще не выполнена).
- "Соединение не установлено" — недоступен WebSocket (прокси/сеть).




## Инструкция для тестера

1. Назначение и область применения
Документ описывает работу тестера в тест-панели и ARM-режиме 'В чем ошибка'. Основной сценарий — проверить одну задачу сразу в нескольких моделях и сравнить ответы по успешности, качеству, скорости, расходу токенов и корректности объяснения.
Инструкция предназначена для QA-специалистов и пользователей, включенных в группу tester. Она помогает проводить сравнение моделей одинаковым способом, чтобы результаты были воспроизводимыми.
Смысл ARM. ARM-тестирование снижает субъективность: одна и та же задача отправляется выбранным моделям, после чего ответы сравниваются по единым признакам.

2. Доступ тестера
Для доступа пользователь должен входить в группу tester. Вход выполняется через страницу /ai/test-panel/login/. После успешного входа тестер автоматически направляется в ARM-раздел 'В чем ошибка'.
    1. Открыть /ai/test-panel/login/.
    2. Войти под учетной записью, которая состоит в группе tester.
    3. Убедиться, что после входа открыт ARM-раздел 'В чем ошибка'.
    4. Если доступ запрещен, проверить членство в группе tester.
Условие	Ожидаемый результат	Если не выполнено
Пользователь в группе tester	Доступ к тест-панели открыт.	Обратиться к администратору прав.
Успешный вход	Автоматический переход в ARM.	Проверить учетную запись и сессию.
Открыт ARM	Доступна форма тестирования.	Проверить маршрут и права.

3. Интерфейс ARM
ARM-раздел предназначен для сравнительной проверки моделей. Форма содержит язык интерфейса, язык программирования, тему, препромпт, условие задачи, код и чекбоксы выбора моделей.
Элемент формы	Назначение	Комментарий тестера
Язык интерфейса	Влияет на формулировку запроса.	Проверяйте, отвечает ли модель на выбранном языке.
Язык программирования	Задает технический контекст.	Обязателен для корректной диагностики кода.
Тема	Уточняет область задачи.	Помогает связать запрос с учебной темой.
Препромпт	Задает методику ответа.	Нужно проверять соблюдение модели инструкций.
Задача и код	Материал для анализа.	Желательно указывать оба поля.
Чекбоксы моделей	Выбор моделей для сравнения.	Чем больше моделей, тем дольше отчет.

4. Подготовка тестового задания
Хороший тестовый кейс должен быть однозначным и воспроизводимым. Тестер должен указать условие, код, ожидаемый результат и тип ошибки. Если в кейсе недостаточно данных, разные модели могут давать разные ответы не из-за качества, а из-за неоднозначности исходных условий.
Компонент кейса	Что включить	Пример
Условие	Что должна делать программа.	Найти максимум в массиве.
Код	Фактический код с ошибкой.	Фрагмент или полный файл.
Ожидаемый результат	Что должно быть на выходе.	Для [1, 5, 2] ожидать 5.
Фактический результат	Что происходит сейчас.	Возвращается 1 или ошибка IndexError.
Ограничения	Что нельзя менять.	Не переписывать алгоритм полностью.

Методическая чистота. Не меняйте задачу между моделями. Для корректного сравнения все модели должны получить один и тот же набор входных данных.

5. Выбор моделей и запуск отчета
Кнопка 'Запустить отчет' запускает последовательную проверку выбранных моделей. Последовательный запуск облегчает контроль ошибок, но увеличивает общее время выполнения при большом числе моделей.
    5. Выбрать язык интерфейса.
    6. Указать язык программирования.
    7. Выбрать тему и препромпт при необходимости.
    8. Заполнить задачу и/или код.
    9. Отметить модели для проверки.
    10. Нажать 'Запустить отчет' и дождаться формирования результатов.
Перед запуском. Убедитесь, что выбранные модели действительно нужны для сравнения. Избыточный выбор увеличивает время и расход токенов.

6. Анализ результатов
После завершения ARM формирует краткий отчет и карточки ответов моделей. Краткий отчет показывает общее число моделей, количество успешных и ошибочных ответов, суммарные токены и самую быструю модель.
Показатель	Что означает	Как использовать
Всего моделей	Сколько моделей участвовало.	Проверить полноту выборки.
Успешно / С ошибкой	Сколько моделей ответило штатно.	Выявить нестабильные или недоступные модели.
Суммарно токенов	Оценка общего объема запроса/ответа.	Контролировать стоимость и размер теста.
Самая быстрая модель	Модель с минимальным временем ответа.	Учитывать в сценариях, где важна скорость.
Карточки ответов	Краткое представление ответа каждой модели.	Открывать модальное окно для полного анализа.

7. Ошибки моделей и человекочитаемые сообщения
Ошибки выводятся в человекочитаемом виде, например таймаут, 401, 429 и другие. Тестер должен фиксировать не только код ошибки, но и условия, при которых она возникла: модель, время, размер запроса, выбранный препромпт и язык.
Ошибка	Возможная причина	Что фиксировать
Timeout	Модель или сеть не ответили вовремя.	Модель, время запуска, размер задачи.
401	Проблема авторизации или токена.	Модель/провайдер, контур, момент возникновения.
429	Лимит или квота провайдера.	Число запусков, время, выбранные модели.
Пустой/неполный ответ	Сбой модели, лимит контекста или обрыв.	Текст запроса, токены, статус.
Ответ не на том языке	Модель проигнорировала язык интерфейса.	Выбранный язык и полный ответ.

8. Ограничения роли тестера
Тестер не видит страницу 'Состояние моделей' и не имеет доступа к управлению справочниками: языками, темами и промптами. Это разделение снижает риск случайного изменения конфигурации во время тестирования.
Доступно тестеру	Недоступно тестеру	К кому обращаться
Запуск ARM-отчета.	Управление справочниками.	Суперадмин.
Сравнение ответов моделей.	Страница состояния моделей.	Суперадмин или администратор проекта.
Фиксация ошибок.	Изменение токенов и инфраструктуры.	Системный администратор.

9. Методика тестирования качества
Качество ответа следует оценивать по нескольким независимым критериям: корректность найденной ошибки, полнота объяснения, минимальность исправления, соблюдение языка, соблюдение препромпта и полезность для пользователя.
Критерий	Высокая оценка	Низкая оценка
Корректность	Модель находит реальную причину ошибки.	Модель исправляет несуществующую проблему.
Объяснение	Причина объяснена пошагово.	Дан только готовый код без объяснения.
Минимальность	Исправляется только нужный фрагмент.	Алгоритм переписан без необходимости.
Соблюдение языка	Ответ на выбранном языке интерфейса.	Ответ смешивает языки или игнорирует выбор.
Соблюдение препромпта	Структура соответствует инструкции.	Препромпт проигнорирован.
Практическая полезность	Пользователь понимает, как не повторить ошибку.	Ответ формально правильный, но не обучает.

10. Оформление дефектов
Если обнаружена проблема, ее следует описывать так, чтобы разработчик или администратор смог воспроизвести ситуацию без дополнительных уточнений.

Шаги воспроизведения	Маршрут, выбранные поля, модель, препромпт, задача и код.
Фактический результат	Что показала модель или система.
Ожидаемый результат	Что должно было произойти.
Вложения	Скриншот, полный ответ модели, время запуска.
Серьезность	Блокер, критичный, средний, низкий — по влиянию на пользователя.

# Инструкция для тестера

Этот документ описывает тест-панель и режим ARM.

## 1. Доступ

- Пользователь должен входить в группу tester.
- Вход осуществляется через страницу:
  - /ai/test-panel/login/
- После входа тестер автоматически направляется в ARM-раздел "В чем ошибка".

## 2. Интерфейс ARM

Раздел: "ARM: В чем ошибка".

Назначение: проверить одну задачу сразу в нескольких моделях и сравнить ответы.

Форма включает:
- Язык интерфейса (влияет на формулировку запроса).
- Язык программирования.
- Тема и препромпт.
- Задача и код.
- Чекбоксы выбора моделей.

Кнопка "Запустить отчет" запускает последовательную проверку выбранных моделей.

## 3. Результаты

- Краткий отчет:
  - Всего моделей.
  - Успешно / С ошибкой.
  - Суммарно токенов.
  - Самая быстрая модель.
- Ответы моделей:
  - Каждая карточка кликабельна.
  - По клику открывается модальное окно с полным ответом.
- Ошибки выводятся в человекочитаемом виде (таймаут, 401, 429 и т.д.).

## 4. Ограничения

- Тестеры не видят страницу "Состояние моделей".
- Тестеры не имеют доступа к управлению справочниками (языки, темы, промпты).




## Инструкция для системного администратора

1. Назначение и контекст инфраструктуры
Документ описывает инфраструктурные требования и порядок запуска AI-приложения на виртуальной машине. Особое внимание уделено работе за корпоративным прокси, Docker/Docker Compose v2, PostgreSQL, портам, WebSocket и эксплуатационной диагностике.
Цель системного администратора — обеспечить воспроизводимый запуск, сетевую доступность сервиса, корректный доступ к внешним API моделей и предсказуемую диагностику при сбоях.
Главная особенность контура. VM не имеет прямого доступа в интернет: внешний трафик, включая docker pull и обращения к API моделей, должен идти через корпоративный прокси.

2. Требования к виртуальной машине
Компонент	Минимальное требование	Комментарий
ОС	Linux, рекомендуется Ubuntu/Debian.	Желательна LTS-версия и актуальные пакеты безопасности.
Сеть	Доступ к интернету через HTTP/HTTPS-прокси.	Проверяется отдельно для shell, Docker daemon и контейнеров.
Docker	Docker Engine и доступ пользователя к Docker daemon.	Обычно требуется членство в группе docker.
Compose	Docker Compose v2.	Проверяется командой docker compose version.
База данных	PostgreSQL локально или внешне.	Внешний PostgreSQL предпочтительнее для отказоустойчивости.
Порты	8080 или 8081 для nginx, 5432 для PostgreSQL при локальной БД.	Порт 8081 указан как production по умолчанию.
Ресурсы	Минимум 2-4 GB RAM и достаточный диск.	Для стабильной работы желателен запас по RAM и месту под логи.

3. Сетевой доступ и корпоративный прокси
Прокси должен быть настроен на тех уровнях, где выполняются сетевые обращения: shell/systemd, Docker Engine, сборка образов и, при необходимости, runtime контейнеров. Наличие прокси только в пользовательской shell-сессии не означает, что Docker daemon сможет скачивать образы.
3.1. Системный прокси
Системные переменные HTTP_PROXY, HTTPS_PROXY и NO_PROXY задают базовые правила выхода в сеть для командной оболочки и сервисов, которые наследуют окружение.
export HTTP_PROXY="http://user:pass@proxy.host:3128/"
export HTTPS_PROXY="http://user:pass@proxy.host:3128/"
export NO_PROXY="localhost,127.0.0.1,db,nginx,.local"

Пароли и спецсимволы. Если в логине или пароле есть специальные символы, их следует указывать в URL-encoded виде, иначе прокси может отклонять запросы.

4. Прокси для Docker Engine и сборки образов
Docker daemon является отдельным системным сервисом. Поэтому для скачивания образов прокси настраивается через systemd drop-in файл.
    1. Создать каталог /etc/systemd/system/docker.service.d.
    2. Создать файл http-proxy.conf с переменными окружения.
    3. Выполнить daemon-reload и перезапустить Docker.
    4. Проверить docker info | grep -i proxy.
sudo mkdir -p /etc/systemd/system/docker.service.d

# /etc/systemd/system/docker.service.d/http-proxy.conf
[Service]
Environment="HTTP_PROXY=http://user:pass@proxy.host:3128/"
Environment="HTTPS_PROXY=http://user:pass@proxy.host:3128/"
Environment="NO_PROXY=localhost,127.0.0.1,db,nginx,.local"

sudo systemctl daemon-reload
sudo systemctl restart docker
docker info | grep -i proxy

4.1. Прокси для build args
При использовании docker-compose.linux.yml переменные HTTP_PROXY и HTTPS_PROXY могут передаваться в секцию build args только на время сборки. Это полезно, если политика безопасности запрещает сохранять прокси в runtime окружении контейнеров.
5. Установка Docker и Docker Compose v2
Рекомендуется устанавливать Docker Engine из официального репозитория Docker, чтобы получить актуальные версии docker-ce и docker-ce-cli. После установки пользователя добавляют в группу docker, затем переподключают сессию.
sudo usermod -aG docker $USER
newgrp docker
docker version
docker compose version

Проверка	Команда	Успешный признак
Docker Engine	docker version	Отображаются client и server версии.
Compose v2	docker compose version	Команда существует и возвращает версию.
Права пользователя	docker ps	Команда выполняется без sudo при принятой политике доступа.

6. PostgreSQL и параметры подключения
PostgreSQL может размещаться на той же VM или быть внешним сервисом. Для production-подхода внешний PostgreSQL предпочтительнее: он проще резервируется, масштабируется и отделяет состояние от контейнеров приложения.
Параметр .env	Назначение	Контроль
DB_HOST	Адрес хоста PostgreSQL.	Доступен из контейнера web.
DB_PORT	Порт PostgreSQL, обычно 5432.	Не блокируется firewall.
DB_USER	Пользователь базы данных.	Имеет права на миграции.
DB_PASSWORD	Пароль пользователя.	Хранится безопасно, не публикуется в логах.
DB_NAME	Имя базы данных.	Существует до запуска миграций или создается регламентно.

Права БД. Пользователь PostgreSQL должен иметь права на создание таблиц и выполнение миграций. Иначе приложение может стартовать, но упасть на этапе инициализации схемы.

7. Подготовка .env и запуск контейнеров
Запуск начинается с подготовки .env. Этот файл должен содержать параметры базы данных, токены провайдеров моделей, сетевые настройки и другие переменные, необходимые приложению.
docker compose --env-file .env up -d --build
docker compose --env-file .env exec -T web python manage.py migrate
docker compose --env-file .env exec -T web python manage.py collectstatic --noinput

# production-режим
AI_NGINX_BIND=127.0.0.1 AI_NGINX_PORT=8081 docker compose --env-file .env up -d --build
docker compose --env-file .env exec -T web python manage.py migrate
docker compose --env-file .env exec -T web python manage.py collectstatic --noinput

Шаг	Действие	Что проверить
1	Создать .env.	Все обязательные переменные заполнены.
2	Запустить docker compose.	Контейнеры собрались и стартовали без критических ошибок.
3	Проверить docker compose ps.	web, nginx и зависимые сервисы находятся в ожидаемом состоянии.
4	Проверить HTTP-доступность.	curl возвращает ожидаемый HTTP-ответ.
5	Проверить WebSocket.	Внешний nginx пропускает Upgrade и Connection: upgrade.

8. Проверка доступности сервиса
После запуска необходимо проверить доступность приложения на уровне HTTP и статических файлов. Это позволяет быстро отличить проблему маршрутизации nginx от проблемы backend-приложения.
docker compose ps
curl -I http://127.0.0.1:8081/ai/chat/
curl -I http://127.0.0.1:8081/ai/static/admin/css/chat_template.css

Проверка	Нормальный результат	Если не работает
docker compose ps	Контейнеры Up/healthy или штатный статус.	Смотреть логи web и nginx.
curl /ai/chat/	HTTP-ответ от nginx/application.	Проверить порт, nginx, upstream.
curl static	Статика отдается.	Проверить collectstatic, volume, nginx location.
WebSocket /ai/chat/ws/	Соединение устанавливается через прокси.	Проверить Upgrade и Connection headers.

9. Эксплуатация на VM
При эксплуатации VM за прокси типовые сбои чаще всего связаны не с кодом приложения, а с окружением: неверный прокси, истекшие токены, лимиты провайдера моделей или блокировка WebSocket внешним nginx.
Симптом	Вероятная причина	Действие
401/403 от моделей	Неверный токен или нет доступа к модели.	Проверить переменные .env и права у провайдера.
429 от моделей	Превышены лимиты или квоты.	Проверить лимиты, расписание тестов, нагрузку.
Timeout	Проблема сети, прокси или провайдера.	Проверить прокси, DNS, маршрутизацию, логи.
WebSocket не подключается	Внешний nginx не пропускает Upgrade.	Добавить заголовки Upgrade и Connection: upgrade.
docker pull не работает	Прокси не настроен для Docker daemon.	Настроить systemd drop-in для docker.service.

10. Диагностика и журналы
Диагностика должна идти от простого к сложному: сначала состояние контейнеров, затем логи, затем сетевые проверки, затем настройки прокси и токенов.
docker compose logs -f web
docker compose logs -f nginx
docker info | grep -i proxy

Источник логов	Что искать	Решение
web	Ошибки приложения, подключения к БД, API моделей.	Исправить .env, миграции, токены или сетевой доступ.
nginx	Ошибки upstream, WebSocket, статика.	Проверить конфигурацию nginx и порты.
Docker daemon	Проблемы pull/build, прокси, DNS.	Проверить systemd proxy и доступность registry.

11. Ротация логов и ресурсы
Для длительной эксплуатации нужно контролировать потребление RAM, CPU, диска и рост логов. Минимум 2-4 GB RAM следует рассматривать как нижнюю границу, а не комфортный запас.
Ресурс	Что контролировать	Профилактика
RAM	OOM, перезапуски контейнеров, деградация скорости.	Заложить запас, мониторить usage.
Диск	Рост docker layers, логов, бэкапов.	Использовать ротацию Docker или logrotate.
CPU	Пики при сборке и нагрузке.	Не запускать тяжелые операции в часы пик.
Логи	Бесконтрольный рост файлов.	Включить ротацию и лимиты.

12. Риски, отказоустойчивость и восстановление
Риск	Последствие	Мера снижения
Единственная локальная БД	Потеря данных при сбое VM.	Внешний PostgreSQL или регулярные бэкапы.

# Сисадмин: инфраструктура и запуск

Этот документ дополняет базовые инструкции по развертыванию и описывает системные требования, прокси и нюансы виртуальной машины.

## 1. Контекст инфраструктуры

- Проект разворачивается на виртуальной машине (VM).
- VM находится за корпоративным прокси, поэтому для доступа к внешним API и скачивания образов нужен прокси.
- Основные зависимости на стороне сервера: Docker, Docker Compose v2 и PostgreSQL.

## 2. Требования к VM

- ОС: Linux (рекомендуется Ubuntu/Debian).
- Доступ к интернету только через прокси (HTTP/HTTPS).
- Открыт доступ к Docker daemon для пользователя (обычно группа docker).
- Свободные порты:
  - 8080 (dev-like) или 8081 (prod по умолчанию) для nginx контейнера.
  - 5432 для PostgreSQL (если БД на той же VM).

## 3. Настройка прокси

### 3.1. Системный прокси (если требуется)

Настройте переменные окружения на уровне shell или systemd:

- HTTP_PROXY
- HTTPS_PROXY
- NO_PROXY

Пример (в bash):

export HTTP_PROXY="http://user:pass@proxy.host:3128/"
export HTTPS_PROXY="http://user:pass@proxy.host:3128/"
export NO_PROXY="localhost,127.0.0.1,db,nginx,.local"

Если в логине/пароле есть спецсимволы, указывайте их в URL-encoded виде.

### 3.2. Прокси для Docker Engine

Если Docker должен выходить в интернет (pull образов), настройте прокси для демона.

1) Создайте каталог:

sudo mkdir -p /etc/systemd/system/docker.service.d

2) Создайте файл /etc/systemd/system/docker.service.d/http-proxy.conf:

[Service]
Environment="HTTP_PROXY=http://user:pass@proxy.host:3128/"
Environment="HTTPS_PROXY=http://user:pass@proxy.host:3128/"
Environment="NO_PROXY=localhost,127.0.0.1,db,nginx,.local"

3) Примените изменения:

sudo systemctl daemon-reload
sudo systemctl restart docker

4) Проверка:

docker info | grep -i proxy

### 3.3. Прокси для сборки образов

Если используется docker-compose.linux.yml, обратите внимание:
- В секции build args можно передавать HTTP_PROXY/HTTPS_PROXY только для сборки.
- В runtime переменные прокси могут быть отключены, если это нужно по политике безопасности.

## 4. Установка Docker и Compose

### 4.1. Docker Engine

Рекомендуемый способ (Ubuntu):

- Установить официальный репозиторий Docker.
- Установить docker-ce и docker-ce-cli.
- Добавить пользователя в группу docker:

sudo usermod -aG docker $USER

После этого перезайдите в сессию или выполните:

newgrp docker

Проверка:

docker version

### 4.2. Docker Compose v2

Проверка:

docker compose version

Если команда отсутствует, установите пакет docker-compose-plugin.

## 5. PostgreSQL

Варианты:
- Локальный PostgreSQL на той же VM.
- Внешний PostgreSQL (рекомендуется для отказоустойчивости).

Минимальные параметры подключения указываются в .env:

- DB_HOST
- DB_PORT (обычно 5432)
- DB_USER
- DB_PASSWORD
- DB_NAME

Убедитесь, что пользователь имеет права на создание таблиц и миграции.

## 6. Запуск контейнеров (рекомендованный порядок)

1) Создать .env и заполнить ключевые переменные.
2) Поднять стек:

docker compose --env-file .env up -d --build
docker compose --env-file .env exec -T web python manage.py migrate
docker compose --env-file .env exec -T web python manage.py collectstatic --noinput

Для прод-режима:

AI_NGINX_BIND=127.0.0.1 AI_NGINX_PORT=8081 docker compose --env-file .env up -d --build
docker compose --env-file .env exec -T web python manage.py migrate
docker compose --env-file .env exec -T web python manage.py collectstatic --noinput

3) Проверить статус:

docker compose ps

4) Проверить доступность:

curl -I http://127.0.0.1:8081/ai/chat/

## 7. Нюансы эксплуатации на VM

- Если VM не имеет прямого доступа к интернету, весь внешний трафик (API моделей, docker pull) должен идти через прокси.
- В случае 401/403/429 от моделей проверяйте:
  - валидность токенов в .env,
  - доступ к конкретной модели у провайдера,
  - корректность прокси (не подменяет ли ответы).
- Для WebSocket /ai/chat/ws/ убедитесь, что внешний nginx пропускает Upgrade и Connection: upgrade.

## 8. Диагностика

- Логи контейнера web:

  docker compose logs -f web

- Логи nginx:

  docker compose logs -f nginx

- Проверка статических файлов:

  curl -I http://127.0.0.1:8081/ai/static/admin/css/chat_template.css

## 9. Ротация и ресурсы

- Убедитесь, что на VM достаточно RAM (минимум 2-4 GB) и свободного диска.
- Для логов контейнеров используйте встроенную ротацию Docker или внешний logrotate.




## Инструкция для суперадмина

1. Назначение и область применения
Настоящая инструкция описывает работу суперадминистратора в административном интерфейсе AI-приложения. Документ раскрывает порядок входа, управление справочниками, жизненный цикл препромптов, глобальное включение AI, контроль состояния моделей и использование ARM-раздела для сравнительного тестирования.
Инструкция предназначена для сотрудников, которым предоставлены расширенные административные права. Ее задача — не только перечислить кнопки интерфейса, но и объяснить логику связей между сущностями, последствия действий администратора и контрольные признаки корректной настройки.
Ключевая идея. Суперадмин управляет не отдельным диалогом пользователя, а конфигурацией всей AI-системы: доступностью интерфейса, набором тем, препромптами и состоянием моделей.

2. Роль суперадмина и границы ответственности
Суперадмин отвечает за корректность данных, которые видят пользователи и тестеры. Ошибка в справочнике или препромпте влияет на качество запросов к модели, а некорректное глобальное выключение AI может остановить пользовательский сценарий полностью.
Объект	Что контролирует суперадмин	Почему это важно
ProgrammingLanguage	Наличие и корректность языков программирования.	Без языка режим решения задач и поиска ошибок может блокировать отправку.
Topic	Связь темы с конкретным языком.	Тема уточняет учебный или технический контекст запроса.

Состояние моделей	Актуальность статусов и ручной запуск проверки.	Пользователи должны выбирать только реально доступные модели.

3. Доступ в административный интерфейс
Административный интерфейс расположен по адресу /ai/admin/. Для доступа требуется учетная запись с признаком staff или superuser. После входа в интерфейс должны быть доступны управление справочниками, ARM-раздел и раздел состояния моделей.
    1. Открыть основной сайт и убедиться, что выполнена авторизация под административной учетной записью.
    2. Перейти по адресу /ai/admin/.
    3. Проверить наличие разделов управления справочниками, ARM и состояния моделей.
    4. При отсутствии разделов проверить права пользователя и членство в соответствующих группах.
Контроль доступа. Если пользователь видит только часть разделов, сначала проверяется роль и набор прав, а не работоспособность приложения. Недостаток прав часто выглядит как отсутствие пункта меню.

4. Управление справочниками
Справочники задают основу пользовательского выбора. Их следует создавать сверху вниз: сначала язык программирования, затем тему, затем препромпт. Такая последовательность исключает ситуацию, когда препромпт создан без понятного контекста.
Сущность	Назначение	Правило заполнения	Пример контроля
ProgrammingLanguage	Список языков программирования.	Название должно быть однозначным и узнаваемым для пользователя.	Python, JavaScript, Java, C++.
Topic	Тема, привязанная к языку.	Тема не должна дублировать язык; она должна уточнять раздел знаний.	Циклы, строки, рекурсия, массивы.
Prompt	Препромпт, привязанный к теме.	Текст должен задавать поведение модели, формат ответа и ограничения.	Объясняй ошибку пошагово, не переписывай код полностью без необходимости.

4.1. Рекомендуемый порядок настройки
    5. Создать или проверить язык программирования.
    6. Создать тему и привязать ее к выбранному языку.
    7. Создать препромпт и привязать его к теме.
    8. Открыть пользовательский режим и проверить, что язык, тема и препромпт отображаются корректно.
    9. Выполнить пробовый запрос и убедиться, что модель получает ожидаемый контекст.
5. Управление препромптами
Препромпт — это управляющая инструкция, которая добавляется к пользовательскому запросу. Он должен быть конкретным, проверяемым и связанным с задачей темы. Хороший препромпт не подменяет пользовательское задание, а уточняет способ ответа модели.
Критерий	Хорошая практика	Плохая практика
Конкретность	Опиши ошибку, причину и минимальное исправление.	Ответь хорошо.
Формат	Дай краткий вывод, затем подробный разбор и исправленный фрагмент.	Без указания структуры.
Ограничения	Не меняй алгоритм, если ошибка синтаксическая.	Всегда переписывай решение полностью.
Язык	Отвечай на выбранном языке интерфейса.	Смешивай языки без необходимости.

Практический смысл. Для учебных задач препромпт должен помогать обучению: объяснять причинно-следственную связь, а не просто выдавать готовый ответ.

6. Глобальное включение и выключение AI
Глобальная доступность AI-интерфейса управляется моделью AIAppSettings и полем is_enabled. Если поле выключено, пользовательские страницы возвращают сообщение 'AI app is disabled'.

Осторожно. Перед выключением AI рекомендуется уведомить заинтересованных пользователей и тестеров, если это не аварийная мера.

Для суперадмина ARM полезен как инструмент контроля качества конфигурации: если часть моделей стабильно отвечает ошибками, проблема может быть не в пользовательском запросе, а в доступности модели, токене, прокси или настройке провайдера.

# Инструкция для суперадмина

Этот документ описывает административный интерфейс, справочники и состояние моделей.

## 1. Вход в админку

- Админка доступна по адресу /ai/admin/.
- Требуется учетная запись staff или superuser.

После входа доступны:
- Управление справочниками.
- Раздел ARM.
- Раздел "Состояние моделей".

## 2. Управление справочниками

### ProgrammingLanguage
- Список языков программирования.

### Topic
- Темы, привязанные к языкам программирования.

### Prompt
- Препромпты, привязанные к темам.
- Отображаются в UI как выпадающий список.

Рекомендуемая последовательность:
1) Язык программирования
2) Тема
3) Препромпт

## 3. Глобальное включение/выключение AI

Модель AIAppSettings:
- Поле is_enabled включает/выключает весь AI-интерфейс.
- При выключении пользовательские страницы возвращают "AI app is disabled".

## 4. Состояние моделей

Раздел: "AI: Состояние моделей" (/ai/admin/arm/models/).

Функции:
- Таблица статусов моделей (Активна/Неактивна).
- Дата окна проверки и время последней проверки.
- Кнопка "Обновить состояние моделей" запускает ручную проверку.
- Таблица обновляется автоматически в фоне.

## 5. ARM-тестирование

Раздел: "ARM: В чем ошибка".

Используется для прогона одной задачи в нескольких моделях и сравнения результатов.

## 6. Плановые проверки моделей

- Плановая проверка выполняется ежедневно в 04:00 МСК.
- Ручной запуск:
  - python manage.py check_models_health
  - python manage.py check_models_health --force
